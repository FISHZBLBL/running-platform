# 2026-07-01 15:23:09 索引文件与前端增量更新

## 改动内容
- 新增用户级 COS 索引 key：
  - `users/{username}/index/runs.json`
  - `users/{username}/index/weights.json`
  - `users/{username}/index/shoes.json`
- 后端列表读取优先走索引文件：
  - 如果索引存在，直接读取一个 JSON 列表。
  - 如果索引不存在，自动从旧的小文件结构恢复列表，并写入索引文件，兼容已有用户数据。
- 新增、编辑、删除跑步记录时同步维护 `runs.json` 索引。
- 新增、编辑、删除体重记录时同步维护 `weights.json` 索引。
- 新增、编辑、删除跑鞋时同步维护 `shoes.json` 索引。
- 删除跑鞋时，后端同步清空关联跑步记录中的 `shoeId`，并更新跑步索引。
- 前端保存/修改/删除跑步、体重、跑鞋后，不再立刻调用全量 `refresh()`。
- 前端预测改为使用共享的 `buildPrediction` 在当前页面状态上实时计算。

## 为什么做这个改动
- 原来的列表接口需要先列 COS 前缀，再逐个读取许多小 JSON 文件。数据越多，首次进入网站越慢。
- 原来的保存流程会在保存一条数据后重新请求跑步、体重、跑鞋和预测，导致保存后等待时间明显。

## 解决的问题
- 首页和历史记录初次加载时，后端读取文件数量从“几十/几百个小文件”降低为几个索引文件。
- 保存、修改、删除单条数据后，页面立即更新当前数据、图表和预测，不再全量重新同步。
- 老用户数据不需要手动迁移，第一次读取会自动生成索引。

## 注意
- 单条完整文件仍然保留，例如 `runs/{runId}.json`，方便以后做详情编辑、截图、分段和备份。
- 索引文件是加速列表读取的聚合数据，不替代单条完整文件。
